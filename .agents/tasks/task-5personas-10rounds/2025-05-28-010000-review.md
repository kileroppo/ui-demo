# Feature-rich UI style gallery overhaul

Ten rounds of persona-driven iteration added dark mode, favorites, a style comparison table, a 5-question recommendation wizard, prompt segmentation into three variants, mobile bottom-tab navigation, chip-based filters with sorting/view-toggle controls, accessibility landmarks, and onboarding tooltips. The approach is hook-per-feature with localStorage persistence, pure-function utilities for scoring and prompt generation, and Tailwind dark-variant classes applied uniformly. Test coverage is reported at 99%+ statements.

Watch for: favorites and compare state are local to their respective component trees and don't synchronize across tabs or even across hook instances within the same render (confirmed). The style advisor scoring function produces ties easily and has no tiebreaker beyond array insertion order (confirmed). The `useThemeMode` hook applies the class both in `useEffect` and synchronously inside `toggle`, creating a double-write on every toggle (confirmed).

## High-level view

Dark mode is implemented via a `dark` class on `<html>` driven by `useThemeMode`, which respects `prefers-color-scheme` as a fallback and persists choice to localStorage. The toggle button is duplicated (desktop nav and mobile header) but both call the same hook — since it uses local state rather than context, they are independent instances that happen to stay in sync only because responsive breakpoints ensure one is always unmounted.

Favorites and onboarding are each a standalone hook backed by localStorage with no shared context. Every component calling `useFavorites()` gets its own state snapshot. The architecture silently breaks if two mounted components both toggle — only the last write wins.

The style advisor is a self-contained page with a linear 5-step wizard that produces scored recommendations via `getRecommendations`. Scoring is additive (no negative weights) and uses hardcoded English keyword lists matched against English-language fields in the dataset, while the UI is entirely Chinese.

Prompt segmentation splits the single Chinese prompt into three variants (UI design, image generation, code generation) via template-string functions in `promptGenerator.ts`.

Mobile UX adds a fixed bottom tab bar (4 items) while the hamburger menu retains all 5 items (adding "About"), meaning "About" is unreachable from the bottom tabs.

The filter panel replaced native `<select>` elements with chip-based buttons. Sorting and grid/list view toggle are new controls; view mode preference persists to localStorage.

Accessibility improvements include `focus-visible` outline via global CSS, ARIA landmarks, `aria-pressed` on toggles, `aria-live="polite"` on result counts, and heading hierarchy verified by a dedicated test suite. The `⌘K` keyboard-shortcut badge in the search bar was missed in the dark-mode pass.

<details>
<summary>Issues (6)</summary>

1. **Favorites hook lacks cross-instance sync** — Multiple simultaneous mounts of `useFavorites()` will overwrite each other's localStorage writes. Wrap in a React context or add a `storage` event listener to sync state.
2. **Theme toggle double-writes DOM class** — `toggle()` calls `applyMode()` synchronously and `useEffect` fires again on the resulting state change. Remove the synchronous call inside `toggle` and let `useEffect` be the single source of truth.
3. **Advisor scoring produces frequent ties with no tiebreaker** — Styles with identical scores appear in dataset insertion order. Add a secondary sort (e.g., alphabetical name) or randomize ties to avoid giving the impression that the first result is clearly better.
4. **Advisor keyword matching is English-only against a Chinese UI** — Scoring uses English keywords ("saas", "portfolio") matched against English fields, but the user sees Chinese labels exclusively. A mismatch between the mental model and the scoring logic could produce puzzling results.
5. **Compare state not persisted or shared** — `useCompare` is ephemeral (no localStorage, no context). Navigating away from the gallery and back loses all selections. Consider persisting to session/localStorage or hoisting into a route-level context.
6. **`kbd` shortcut hint has insufficient dark-mode contrast** — The `⌘K` badge in `SearchBar` uses hardcoded `text-gray-500 bg-gray-100 border-gray-200` without dark variants, producing low-contrast text on dark backgrounds.

## Favorites hook state isolation

`useFavorites` initializes from `localStorage` on mount and writes back on every toggle, but holds state in `useState`. If two components mount with this hook (e.g., `StyleCard` and the gallery's favorites-count badge both calling `useFavorites`), they read the same initial value but maintain independent copies. A toggle in one won't reflect in the other until a re-mount. In the current tree, `Layout` calls `useFavorites()` for `count` (nav badge) and each `StyleCard` calls it for the heart button — the nav badge's hook instance won't see an updated count until a re-render is triggered by something else, unless a `storage` event or context bridges them.

The same isolation applies to `useOnboarding` and `useThemeMode`, though theme is less dangerous because responsive breakpoints ensure only one toggle instance is mounted at a time.

## Theme toggle double-apply

```ts
const toggle = useCallback(() => {
  setModeState((prev) => {
    const next = prev === 'light' ? 'dark' : 'light'
    localStorage.setItem(STORAGE_KEY, next)
    applyMode(next)  // <-- synchronous DOM mutation
    return next
  })
}, [])
```

The `useEffect` also calls `applyMode(mode)` on every state change. Every toggle triggers class manipulation twice: once inside the updater, once when the effect fires. Currently harmless (classList.add is idempotent), but if `applyMode` ever acquired side effects beyond classList this would double-fire them. The fix is to remove the synchronous call and let `useEffect` be the single source of truth.

## Style advisor scoring semantics

The scoring function awards 0-4 points per dimension with no negative weights. With only 5 questions and small point ranges, ties are common. Results are `scored.slice(0, 5)` after descending sort, but ties within that top-5 fall back to dataset insertion order — first defined wins, giving an arbitrary impression of ranking.

The keyword-matching uses English tokens (`"saas"`, `"creative"`, `"portfolio"`) against the English-language `bestFor` field, while the user interacts entirely in Chinese. A user choosing "创意设计" and seeing results that matched against `"creative"` in a hidden English field might not understand the ranking.

## Prompt segmentation edge case

If a style's `implementationChecklist` is empty, the code-gen prompt ends with "实现清单：。" (dangling punctuation from `join('；')` on an empty array producing an empty string). The test suite uses a style with a non-empty checklist, so this path isn't covered.

## Compare state ephemeral by design

`useCompare` holds selection in `useState` with no persistence. The `CompareBar` constructs `/compare?ids=1,2,3` so the comparison page itself survives reloads, but navigating away from the gallery resets the selection dock. If a user browses a style detail and returns, their compare picks are gone.

## Mobile navigation asymmetry

The bottom tab bar shows 4 items (Home, Styles, Advisor, Products). The hamburger dropdown shows all 5 (adds "About"). "About" is only reachable via hamburger on mobile, which may confuse users who rely exclusively on the bottom tabs.

## Keyboard shortcut badge contrast gap

```tsx
<kbd className="... text-gray-500 bg-gray-100 rounded border border-gray-200">
```

No `dark:` variants. On a dark background (`dark:bg-gray-800` on the input), this badge produces a light rectangle with low-contrast text and an invisible border.

## Accessibility gaps not covered by tests

The dedicated test suite verifies landmarks, skip-link, heading hierarchy, `aria-live`, and interactive labels. Not tested: focus-trap inside the mobile menu (no Escape-to-close), color contrast ratios, and keyboard navigation through chip-based filters.
