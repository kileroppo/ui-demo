# Chinese UI Style Showcase Gallery

A greenfield React application that presents 76 UI design styles as a browsable, searchable gallery with live CSS demos, Chinese translations, AI prompt text, and product-type recommendations. Built on React 19 + Vite 6 + Tailwind CSS v4 + TypeScript, with code splitting via React.lazy and a bilingual (Chinese/English) search engine using alias expansion. The test suite reaches 99.84% statement coverage across 162 tests with enforced 90% thresholds.

**Watch for:** Large static data files (11.5K lines) bundled as TypeScript literals will inflate the client bundle and block tree-shaking (confirmed). The Chinese prompt generation in `build-data.mjs` truncates fields by character count without regard for word/sentence boundaries, producing grammatically broken Chinese text (confirmed). Search operates on the full style array on every keystroke with no debounce (confirmed).

## High-level view

The application is a single-page gallery organized into five routes (home, style list, style detail, product recommendations, about) with lazy-loaded pages behind a shared Layout shell. Navigation is straightforward SPA routing with URL state sync for filters. The architecture is clean for a showcase app of this scope.

Data is generated offline from CSV files by a Node build script (`build-data.mjs`) that produces large TypeScript array literals. These files total 11.5K lines and ship entirely to the client regardless of which styles are actually rendered. The colors and typography datasets are exported but never consumed by any component, making them pure dead weight.

Search combines Chinese alias expansion (18 terms) with full-text matching across multiple fields. It works well for the demo's scale but has no input debounce, meaning every keystroke recalculates against all 76 styles. At this dataset size the performance impact is negligible, but the pattern sets a bad precedent if the dataset grows.

The Chinese prompts are machine-assembled from field substrings with hard character cuts. The results read like truncated sentences ("适用于：Enterprise apps, dashboards, documentation sites, ") with trailing commas and mixed English/Chinese. This undermines the stated goal of providing "即用型中文AI提示词."

Test coverage is genuinely high (99.84% statements, 95.97% branches) and the tests exercise real user interactions through Testing Library, not just render smoke tests. The only material gap is the `build-data.mjs` script which has zero test coverage and contains a hand-rolled CSV parser with edge-case risks.

<summary>Issues (7)</summary>

1. **Dead data exports** — `colors` and `typography` are exported from `src/data/index.ts` but never imported by any component. Remove them or build the pages that use them to avoid bundling ~4.9K lines of unused data.
2. **No bundle splitting for data** — All 76 styles are imported synchronously in every page that references `styles`. Move to dynamic import or a JSON fetch to allow the data to live in a separate chunk.
3. **Truncated Chinese prompts** — `generatePromptZh` in `build-data.mjs` slices strings at arbitrary character positions, producing broken Chinese sentences with trailing commas and mid-word cuts. Truncate at sentence or clause boundaries instead.
4. **No search debounce** — `StyleGallery` and `HomePage` run `searchAndSort` on every keystroke. Add a 150-300ms debounce to `handleQueryChange` to avoid unnecessary re-renders as the dataset grows.
5. **Untested build script** — `build-data.mjs` contains a custom CSV parser with quote handling and is the sole source of all runtime data, yet it has no tests. A malformed CSV row silently produces empty fields.
6. **`category` vs `type` field confusion** — `UIStyle` has both `category` and `type` fields, the build script sets them to the same value, and the UI only ever reads `type`. The `category` field is dead weight that may mislead future developers.
7. **Product `nameZh` fallback to English** — When a product name has no entry in `productZhNames`, the build script falls back to the English name (e.g., "E-commerce" appears as `nameZh` for id=3). This breaks the Chinese-friendly premise.

## Data pipeline and bundle weight

The offline build script (`scripts/build-data.mjs`) reads CSV files from a sibling repository and emits TypeScript array literals. The custom CSV parser handles quoted fields and escaped double-quotes but has no validation layer: a missing column in the source CSV silently produces an empty string, and there's no schema check. The script is run manually and its output is committed directly; there's no CI step to regenerate or verify freshness.

The resulting data files are large:

```
styles.ts      3,329 lines
products.ts    3,254 lines
colors.ts      3,385 lines
typography.ts  1,533 lines
```

The `colors` and `typography` modules are exported from `src/data/index.ts` but imported by zero components. They still end up in the bundle because Vite will include any module that's part of the dependency graph via barrel exports. This adds ~4.9K lines of unused data to the client payload.

Even for the actively-used `styles` and `products`, the import pattern (`import { styles } from '../data'`) pulls the entire dataset synchronously into whichever chunk first references it. With React.lazy splitting the pages, the styles array likely lands in a shared chunk that loads on any navigation. For 76 styles this is tolerable (~100KB gzipped estimate), but the architecture doesn't scale. A JSON fetch or dynamic import would allow the data to be a separate cacheable chunk.

## Chinese prompt quality

The `generatePromptZh` function in the build script assembles prompts by concatenating fixed Chinese labels with substring slices of English-language field values:

```javascript
`请设计一个${name}界面。使用：${effects.substring(0, 60)}。配色：${colors.substring(0, 50)}。适用于：${bestFor.substring(0, 50)}。技术要点：${css.substring(0, 60)}。`
```

The character-count truncation produces results like:

> "适用于：Enterprise apps, dashboards, documentation sites, 。技术要点：display: grid, gap: 2rem, font-family: sans-serif, color: #0。"

Trailing commas, mid-word hex color cuts ("#0"), and English field content inside Chinese sentence frames. The prompts are presented to users as "AI 提示词一键复制" — copy-paste text intended for AI tools. Users copying these will get malformed prompts that require manual cleanup, defeating the one-click value proposition.

## Search architecture

The search system is well-designed for its current scale. `searchStyles` expands Chinese query terms through an alias map of 18 entries, then checks whether all original terms match against a concatenated searchable-text blob per style. `searchAndSort` adds relevance scoring that boosts exact name matches above keyword matches.

The concern is execution frequency: both `HomePage` and `StyleGallery` call `searchAndSort` on every keystroke (via `onChange` → `setQuery` → `useMemo` recalculation). There's no debounce. With 76 styles the cost is sub-millisecond, but the URL state update (`setSearchParams`) also fires per keystroke, pushing history entries. The `{ replace: true }` option in `StyleGallery` prevents history pollution, which is good — but `HomePage` doesn't sync to URL at all, creating an inconsistency where search state is lost on navigation from the home page.

## Component design and accessibility

The component hierarchy is clean: Layout wraps all routes, pages own their state, and presentational components (StyleCard, StyleDetail, FilterPanel) are stateless. Accessibility is above average for a demo project: skip-to-content link, `aria-label` on navigation regions, `aria-live="polite"` on result counts, `aria-current="page"` on active nav items, and keyboard shortcut (Cmd/Ctrl+K) for search focus.

The mobile nav implementation closes on link click (preventing the common "stuck open menu" bug). The `aria-expanded` attribute on the hamburger button correctly reflects menu state.

One gap: the `StyleCard` has `focus-within:ring-2` for keyboard navigation, but the card itself isn't a focusable element — only the inner link is. A keyboard user tabbing through the grid will focus the link, triggering the focus-within ring on the card, which works visually but the card's hover styles (translate, shadow) don't activate on keyboard focus. Minor, but inconsistent with the mouse experience.

## Test quality assessment

The test suite goes beyond render-and-assert. Tests use `userEvent` for realistic interaction simulation (typing, clicking, selecting options). The clipboard utility tests mock at the navigator API level and cover both the modern clipboard API path and the legacy `execCommand` fallback. The search tests verify Chinese alias expansion, multi-term AND logic, and relevance scoring.

What's genuinely tested:
- User interactions: typing in search, selecting filters, toggling mobile menu, copying text
- Edge cases: empty queries, no-match states, maximum keyword display limits, clipboard failures
- Routing: lazy-loaded pages resolving, breadcrumb navigation, 404 handling for invalid style IDs
- Accessibility: aria labels present, skip link rendered, aria-current on active nav

What's not tested:
- The `build-data.mjs` script (CSV parsing, field mapping, Chinese name lookup, prompt generation)
- The `getCategoryCounts` helper in `StyleGallery.tsx` (lines 102-107, uncovered per coverage report)
- URL state synchronization in `StyleGallery` (the `updateSearchParams` callback is exercised indirectly but the actual URL params aren't asserted)
- The `colors` and `typography` data modules (imported but never rendered, so coverage shows 100% vacuously)
- Any integration test that verifies the full data pipeline produces valid runtime data

The 99.84% coverage number is real but slightly inflated by the large data files (3K+ lines each) that are 100% covered simply by being imported — they contain no logic, just array literals.

## `category` vs `type` field redundancy

The `UIStyle` interface declares both `category` and `type`. The build script sets them to the same CSV column value (`s['Type']`). The UI exclusively reads `type` for display and filtering. `category` is included in the searchable text blob (in `searchStyles`) but provides no additional signal since it's identical to `type`. This redundancy adds confusion for any developer extending the data model — it's unclear which field is canonical.
